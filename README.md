1. Multi-Agent Architecture
The system consists of three specialized agents orchestrated sequentially using LangGraph:
Intake Agent: Responsible for receiving and validating user input. It sanitizes customer and merchant IDs, enforces format rules, checks for prompt injection attempts, and applies rate limiting. If input is invalid, the pipeline terminates here and no LLM call is made.
Analysis Agent: Receives the validated input, performs a graph lookup on the fraud transaction network to retrieve risk score, transaction amount, and category, then calls the Gemini LLM to generate a natural language explanation of the risk signals. The LLM is strictly scoped to explanation so it does not make decisions.
Decision Agent: Receives the enriched state from the Analysis Agent and applies rule-based logic to produce a final decision whether it’d be APPROVE, FLAG FOR REVIEW, or BLOCK, based on risk score thresholds and fraud flags.
Agents communicate through a shared typed state dictionary passed between LangGraph nodes. The system operates sequentially with a conditional edge after the Intake Agent. If an error is detected, the graph routes to END, preventing unnecessary LLM calls. This design reflects a hierarchical pattern where each agent has a single responsibility and clear boundaries.
2. Security, Safety, and Guardrails
Input validation is handled in the Intake Agent using regex sanitization, allowing only alphanumeric characters. A pattern-matching list blocks known prompt injection attempts such as "ignore previous instructions" or "act as" before input ever reaches the LLM. Length limits also help prevent abuse through oversized inputs. Rate limiting is implemented using a per-session request log, capping requests at 10 per minute to prevent API abuse during live deployment.
LLM guardrails are enforced in the Analysis Agent. Data values are sanitized before being injected into the prompt, and the prompt explicitly instructs the model to only discuss the transaction and ignore any instructions in the data fields. The LLM is constrained to a single role of just explanation and never controls the final decision.
The API key is stored as a Streamlit Cloud secret and loaded via an environment variable (never hardcoded or logged for security reasons obviously). No personally identifiable information beyond synthetic transaction IDs is processed by the system.
3. Implementation Approach
BFS finds the shortest path by edges and ignores risk. A* finds the lowest cost path guided by fraud risk scores. Comparing them shows whether risk-aware search actually finds more suspicious transaction paths than blind search. I built the underlying fraud graph with BFS/A* search and also performed sensitivity analysis to understand the system's behavior under uncertainty. Since ML risk scores are probabilities and not guaranteed to be correct, the system needs to be evaluated under realistic conditions. Edge weight perturbation adds random noise to transaction costs to simulate a fraud model that is not perfectly accurate, and measures how stable each algorithm's results are. Node removal simulates blocking high-risk customers or merchants from the network and tests whether paths can still be found when those entities are gone. The risk threshold sweep changes the cutoff used to decide what counts as a high-risk transaction, showing how sensitive the system is to that decision boundary. Monte Carlo simulation runs hundreds of perturbed versions of the graph and measures how much the results vary across all of them using a metric called the Coefficient of Variation. Together these four experiments show that BFS is stable but ignores risk completely, while A* is risk-aware but sensitive to noisy predictions. That finding is exactly what motivated the multi-agent layer, where the LLM explains the risk reasoning in plain English so a human can check whether it makes sense before acting on the decision.
That analysis showed A*'s performance depends heavily on risk score quality which motivates the multi-agent layer, where the LLM explains the risk reasoning so a human reviewer can validate it before acting on a decision.
The system is built in Python using LangGraph for agent orchestration, LangChain Google GenAI for Gemini API integration, and Streamlit for the user interface. The underlying fraud data layer uses a NetworkX transaction graph as it natively supports directed graphs with BFS and A* search algorithms and a Random Forest risk scoring model.
Agents are instantiated as Python functions and registered as nodes in a LangGraph StateGraph. The graph is compiled once at startup and reused across requests. Agent termination is handled by LangGraph's END node. Error handling is built into the state where each agent checks for an existing error key before executing, allowing for a graceful pipeline termination without exceptions.
Testing was performed locally by running the full pipeline against real customer and merchant IDs from the BankSim dataset found on Kaggle, verifying correct routing, LLM responses, and decision outputs across low, medium, and high risk transactions.
4. Use of AI/LLMs and Collaboration
The Gemini LLM is used exclusively in the Analysis Agent for natural language explanation of fraud risk signals. It receives transaction attributes of amount, category, risk score, fraud flag and generates a 2-3 sentence summary of what the risk level suggests. Agents collaborate through shared state rather than direct negotiation. Each agent enriches the state and passes it forward. The Intake Agent ensures data quality, the Analysis Agent adds intelligence, and the Decision Agent enforces policy.
The key design tradeoff is autonomy vs. control which is an accurate reflection of the conflict around integrating AI in society. The LLM is given narrow autonomy by simply explaining the transaction while control over the actual fraud decision remains with deterministic rule-based logic which aligns with traditional fraud analysis systems. This prevents the model from making consequential financial decisions autonomously, which aligns with responsible AI deployment principles. To take it a step further in a production environment, this architecture would extend to AWS EC2 for hosting and AWS Bedrock for LLM access, consolidating the entire stack within a single cloud boundary.
